# MASTER-SPEC: RenderCV-AG (Antigravity Sastrería Fork)

> **Propósito:** Actuar como el motor central agnóstico de compilación y orquestación dinámica para la Máquina de Sastrería de CVs, permitiendo el despliegue automático de Currículums hiper-focalizados listos para evadir filtros ATS.

## 1. El Propósito Guía (Outcome)

Convertir los procesos de postulación lineal y manual humana en un flujo algorítmico asíncrono y volumétrico. El objetivo es que la IA actúe como "Matchmaker", destilando ofertas laborales crudas y conectándolas dinámicamente con una Base de Conocimientos, pariendo iteraciones curriculares perfectas (PDFs) al instante, en el estilo visual adecuado, para anular la barrera de entrada humana ante portales corporativos.

## 2. Definición del Artefacto (Output)

- **Motor CLI (Core):** Un fork intervenido de RenderCV de uso local que obedece a instrucciones nativas en formato YAML para renderizados ultrarrápidos, anulando dependencias perimetrales.
- **Base de Conocimiento (Knowledge Base):** Una estructura de directorios en Markdown estandarizado (`experience.md`, `skills.md`, `extracurricular.md`) diseñada bajo preceptos granulares ("Technical Stack") para análisis de inmersión de IA. Cada entry contiene `Details` (prosa descriptiva completa), `Condensed Bullets` (exactamente 3 versiones comprimidas CV-ready en español con nominalización) y `Condensed Bullets EN` (exactamente 3 versiones en inglés con action verbs en pasado). El agente usa los Condensed Bullets como fuente primaria según el idioma del template. Los proyectos extracurriculares incluyen `Subtitle` (mapea al campo `detail` del YAML) y opcionalmente `URL` (mapea a hipervínculo markdown en `name`). El subdirectorio `raw dumps/` contiene las fuentes de verdad factual originales (LinkedIn, CV PDF) contra las cuales se verifica la integridad del contenido extraído.
- **Templates Base Bilingües:** Dos lienzos YAML que funcionan como base obligatoria de toda generación de CV: `default_template.yaml` (español, `locale: spanish`) y `default_template_english.yaml` (inglés, `locale: english`). Ambos contienen datos hardcodeados del candidato (nombre, contacto, educación INTOCABLE, headline inmutable, tema `jpmr`, bloque `settings`). El template puede llegar vacío o parcialmente pre-rellenado por el usuario desde el frontend. El agente aplica el Protocolo de Gap-Fill (`rules/03-gap-fill-protocol.md`) para clasificar cada campo en 4 estados (VACÍO, FIJADO, PARCIAL, INSTRUIDO) y rellena únicamente los gaps sin tocar contenido fijado. Un **Protocolo de Sincronización** (`rules/04-template-sync-protocol.md`) garantiza coherencia estructural entre ambos templates comparando `mtime`; el más reciente actúa como fuente de verdad para resolver discrepancias.
- **Directorio de Temas Centralizados (`themes/`):** Contenedor aislado e independiente de las maquetas bases, alojando diseños personalizados con IDs en minúsculas estrictas para evadir fallos de compilación strict (ej. `jpmr`).
- **Agentes (Antigravity / Kairós):** Implementaciones bajo la carpeta `.agent` conteniendo reglas, workflows y destrezas para gobernar cómo interactúa la IA iterando versiones a medida.
- **Web UI & Dynamic API:** Interfaz React (`rendercv-web`) apalancada por un backend FastAPI (`web/api`). La arquitectura visual consulta iterativamente los recursos (ofertas, CVs, themes y fuentes) haciéndolo hiper-resiliente frente a eliminaciones en crudo de ficheros, y proveyendo pre-visualizaciones en vivo de los YAML inyectados por la IA. El visor PDF implementa una arquitectura de **doble-renderizado con CSS Grid stacking** que mantiene el documento anterior visible mientras el nuevo se dibuja en una capa invisible (`opacity: 0`), ejecutando un swap atómico solo cuando todas las páginas han completado su `onRenderSuccess`, eliminando todo parpadeo visual. El PDF se rasteriza a un **ancho fijo de 800px** y CSS `width: 100% !important` en el canvas (clase `.pdf-page-scaled`) escala el resultado para llenar siempre el espacio horizontal del panel sin re-rasterización ni flicker al redimensionar el divisor. Los hipervínculos del PDF se abren en nueva pestaña (`externalLinkTarget="_blank"`). La Toolbar incluye: selector de ofertas recientes con **botón Save** (💾, `PUT /api/ofertas/{filename}`) para sobreescribir YAMLs desde el editor, selector de tema y fuente (con fuentes bundled de RenderCV vía `/api/fonts`), **dropdown "Estilo"** (`max-w-[90vw]` responsive) con 7 campos configurables por-field (Cuerpo, Nombre, Titular, Contacto, Secciones, Subsecciones, Sub. detalle) cada uno con **3 sliders**: tamaño, peso (100-900, step 50) y **espaciado** (cm, 0.05–2.0, step 0.05). Todos los valores numéricos son **editables por teclado** con **clamping automático** al rango definido. Los sliders usan CSS custom (`appearance: none` + track/thumb por clase `slider-size/weight/spacing`) para garantizar colores consistentes en dark mode. Botón "Reiniciar" restaura los últimos defaults guardados desde `themeDefaults` en memoria. Botón **"Guardar"** persiste los valores (incluyendo spacing) como nuevos defaults via `PUT /api/theme-defaults/{theme_id}` (sidecar JSON en `themes/.overrides/`). El endpoint GET acepta `?fresh=1` para obtener defaults originales del theme ignorando overrides. Feedback visual ✓ verde en ambos botones. **Dropdown "Plantilla base"** con opciones 🇪🇸 Español y 🇬🇧 Inglés que carga el template correspondiente vía `GET /api/default-template?lang=es|en`, y botón condicional **"Actualizar plantilla (Español/Inglés)"** que sobreescribe el template del idioma activo vía `PUT /api/default-template?lang=es|en`. Botón **"Save in CVs"** que renderiza el CV actual y lo guarda directamente en `CVs/Martin Gil CV - [Cargo] - [Idioma].pdf` vía `POST /api/save-cv`, extrayendo el cargo del `output_folder` del YAML y el idioma de `locale.language`, con estados visuales de loading/success/error. El tema seleccionado y los pesos persisten en `localStorage`. Los paneles del editor y el visor se separan mediante un divisor arrastrable (`split-pane-divider`). Todo el Frontend posee Test-Driven Development (TDD) en Vitest y End-to-End con Playwright sobre Chromium.

## 3. Requisitos de Usuario

- El usuario puede pegar una "Oferta de trabajo en crudo", detonando la generación automática y el guardado orgánico dentro de la carpeta `CVs/`.
- El usuario puede modificar manualmente el YAML generado, aplicando cambios reflectivos directos en el PDF y moviendo este último al directorio final, usando el watcher interno y sin dependencias Node para esta tarea específica.
- El usuario puede alimentar la Base de Conocimientos con URLs extrañas, y una habilidad auto-recabará las dependencias detrás de estos proyectos para granularizarlas en la matrix.

## 4. Límites y Restricciones (Límites Intransgredibles)

- **Prohibición de Eliminación de Conocimiento:** (Regla Append-Only). Todo el conocimiento agregado a la `knowledge_base/` *nunca* puede ser reducido ni reemplazado; solo expandido. Correcciones factuales (verificadas contra `raw dumps/`) sí están permitidas.
- **Prohibición de AI-Tells:** Reglas estrictas de sigilo sintáctico que anulan la generación de verbos subjetivos, viñetas débiles, buzzwords en inglés y español, y em-dashes/dobles guiones (`—`, ` -- `) como marcador estilístico de IA.
- **Prohibición de Parches de Entorno:** El proceso de compilación y orquestación debe depender 100% en las capabilities del Fork CLI (usando el payload interno `RenderCommand` y `--watch`). Nada de scripts demonizados espiando carpetas con `chokidar`.
- **Prohibición de Inyección de JD:** Los bullets de experiencia reflejan EXCLUSIVAMENTE hechos de la `knowledge_base/`. El JD se usa para seleccionar contenido, nunca para reescribir lo que el candidato hizo.
- **Headline Inmutable:** El headline del `default_template.yaml` no se modifica por el agente bajo ninguna circunstancia, salvo instrucción explícita del usuario en el mismo mensaje.
- **Inmutabilidad de Contenido Fijado (Gap-Fill Protocol):** Cuando el template llega pre-rellenado, el agente tiene prohibido parafrasear, reformular, eliminar o reordenar cualquier contenido fijado por el usuario. Solo puede rellenar los slots vacíos, interpretar instrucciones entre corchetes `[...]`, y agregar contenido complementario. El agente SÍ puede recomendar mejoras al contenido fijado en su informe de entrega, siempre en función de la oferta laboral específica, pero NUNCA ejecutarlas en el YAML.
- **Soberanía Estructural del Template:** El `default_template.yaml` es la ÚNICA fuente de verdad para la estructura del CV. Los nombres de secciones, la cantidad de entries por sección y la cantidad de highlights por entry los define el template, no el agente. El agente no puede crear, eliminar ni renombrar secciones que no existan en el template. La regla de hipervínculo markdown en proyectos con URL documentada en la KB se mantiene como única directiva de formato.
- **Anti-Solapamiento Experiencia ↔ Proyectos Personales:** Un proyecto personal NO puede incluirse en la sección `Proyectos Personales` del CV si su contenido sustancial ya está cubierto por un entry en `Experiencia`. Ante solapamiento (mismo nombre, misma empresa o ≥50% de overlap temático), la experiencia prevalece y el proyecto se descarta. Art. 5 de `02-cv-sartorial-crafting.md`.
- **Condensación Obligatoria Bilingüe (Art. 6):** Los bullets del CV NUNCA deben copiarse textualmente desde los `Details` de la KB. Cada entry de la KB contiene exactamente 3 `Condensed Bullets` (español, nominalización) y 3 `Condensed Bullets EN` (inglés, action verbs en pasado) pre-comprimidos y CV-ready. El agente los usa como fuente primaria según el idioma del template, recurriendo a `Details` solo como fallback (con compresión agresiva). Al crear o regenerar bullets, el agente DEBE generar AMBAS versiones simultáneamente. Una sola capa de nominalización (ES) o un solo action verb (EN); nombrar herramientas, no describir funciones.
- **Regla Verbal Bilingüe (Art. 2 bis):** Si `locale: spanish`, obligatorio usar nominalización ("Desarrollo de...", "Implementación de..."). Si `locale: english`, obligatorio usar past-tense action verbs sin sujeto explícito ("Developed X", "Implemented Y"). Prohibidos: gerundios, voz pasiva, nominalizaciones en EN, primera persona, pronombres.
- **Acrónimos de Industria (Art. 7):** Acrónimos estándar (AWS, GCP, CI/CD, API, REST, SQL, LLM, NLP, RAG, ML, K8s, etc.) NUNCA deben expandirse en el CV. Se usan siempre en forma corta, incluso si la oferta usa la forma larga.
- **Cargo Sin Nivel de Experiencia:** Al extraer el cargo de una JD para nombrar archivos (`output_folder`, nombre de oferta, PDF guardado), NUNCA incluir calificadores de nivel (Senior, Junior, Lead, Staff, Principal, Mid-Level, Entry-Level, Intern, Trainee, Associate). Solo el rol funcional. Enforced en 3 capas: instrucciones IA (`tailor-cv.md`, `SKILL.md`) y regex safety net (`main.py`).
- **Validación Obligatoria Pre-Entrega:** El agente tiene PROHIBIDO entregar un YAML de CV al usuario sin antes validarlo contra el parser real de RenderCV (`scripts/validate_yaml.py`). Si la validación falla, debe corregir y re-validar en loop hasta obtener exit 0. Protocolo completo en `rules/05-mandatory-validation-protocol.md`, gate de calidad integrado en `tailor-cv.md` (Paso 4.5).

## 5. Prioridades y Trade-offs

- **Soberanía y Rapidez de Generación VS Simplicidad de Setup:** Renunciamos a utilizar templates convencionales de LaTeX para apalancarnos enteramente en la capa determinista en Typescript/Python/YAML, habilitando intervención por Modelos Generativos 10 veces más veloces y modulares con menor índice de fallos estructurales.
- **Inyección YAML VS Scripting Múltiple:** Intervenimos el código nativo de RenderCV desde su payload dict (`settings`) antes que programar macros o scripts mutantes en Bash; preferimos sacrificar pureza del repositorio base por absoluta robustez ejecutora.
- **Desacoplamiento API Frontend vs Hardcodeo React:** Se decidió sacrificar el tiempo inicial para programar parsers en FastAPI que analicen dinámicamente el file-system (`ofertas/`, `themes/`, `CVs/`) en pos de construir una UI en React total y absolutamente tolerante a fallos si un usuario manual o la IA elimina estructuras de disco duro de golpe. La UI no debe crashear, debe volverse silenciosa o advertir.
